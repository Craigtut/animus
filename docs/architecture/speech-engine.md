# Shared Speech Engine Architecture

> **See also**:
> - `docs/architecture/voice-channel.md` -- Voice channel adapter, frontend UX, audio pipeline, streaming pipeline for voice
> - `docs/architecture/tts-licensing-and-distribution.md` -- TTS model licensing (CC-BY-4.0), redistribution, attribution, voice cloning consent

How Animus hears and speaks -- shared STT and TTS as backend infrastructure modules, not channel-specific. Multiple consumers (voice channel, reflex system, agent MCP tools, plugins) access speech capabilities through a unified service facade.

## Overview

The speech engine extracts STT and TTS from the voice channel adapter into shared backend infrastructure. Rather than the voice channel owning its own recognizer and synthesizer instances, all speech capabilities are centralized in a singleton `SpeechService` that any part of the system can use.

**Consumers:**
- **Voice channel adapter** -- transcribes incoming audio, synthesizes replies
- **Reflex system** -- streams TTS for fast voice replies
- **MCP tools** -- `transcribe_audio` and `generate_speech` tools available to the mind and sub-agents
- **Plugins** -- any plugin can access speech via the service singleton

## Technology Choices

### Speech-to-Text: Parakeet TDT v3 via sherpa-onnx

**Model**: NVIDIA Parakeet TDT 0.6B v3 (int8 quantized)
**Runtime**: sherpa-onnx (native Node.js addon via npm)

Parakeet TDT v3 is a 600M-parameter transducer-based ASR model from NVIDIA that achieves state-of-the-art accuracy while running efficiently on CPU. It supports **25 European languages** with automatic language detection. The int8 quantized ONNX version runs locally through sherpa-onnx without internet access.

| Criterion | Value |
|-----------|-------|
| **Accuracy** | 9.7% average WER across 25 languages |
| **Multilingual** | 25 European languages with automatic language detection |
| **Speed** | Faster than real-time on CPU via ONNX Runtime |
| **Local** | Completely offline -- no API calls, no internet needed |
| **Node.js native** | sherpa-onnx npm package with native addon bindings. No Python. |
| **Model size** | ~630 MB (int8 quantized encoder + decoder + joiner) |
| **Input** | 16kHz mono PCM audio |
| **License** | CC-BY-4.0 |

### Text-to-Speech: Pocket TTS via @animus-labs/tts-native (napi-rs)

**Model**: Pocket TTS (~225MB safetensors)
**Runtime**: `@animus-labs/tts-native` -- a napi-rs native Node.js addon wrapping the [pocket-tts](https://github.com/babybirdprd/pocket-tts) Rust crate (Candle-based inference)

Pocket TTS is a zero-shot voice cloning model that uses reference audio to reproduce any voice. Given a short WAV sample (5-15 seconds), it generates speech that mimics the speaker's voice characteristics. The Rust port runs the original safetensors weights natively via Candle and produces near-identical quality to the original Python implementation at 3-7x faster speeds.

**Why native Rust instead of ONNX?** All available ONNX exports of Pocket TTS produce garbled audio. This was thoroughly tested in Feb 2026 (see [ONNX Investigation](#onnx-pocket-tts-investigation) below). The Rust port bypasses ONNX entirely, running the original safetensors weights through Candle for correct, high-quality output.

| Criterion | Value |
|-----------|-------|
| **Runtime** | `@animus-labs/tts-native` napi-rs addon (Rust, compiled per-platform) |
| **Voice cloning** | Zero-shot cloning from 5-15 seconds of reference audio |
| **Speed** | ~3.2x realtime on Apple Silicon (e.g., 5.2s audio in 1.6s) |
| **Quality** | Near-identical to original Python Pocket TTS implementation |
| **CPU-only** | No GPU needed. Uses Accelerate framework on macOS. |
| **License** | CC-BY-4.0 (model weights), MIT (Rust crate) |
| **Model size** | ~225 MB (safetensors + tokenizer) |
| **Build requirement** | Rust toolchain required for first build (~2 min). Auto-built on `npm run dev`. |

## Module Structure

```

  audio-utils.ts     -- webmToPcm, pcmToWav, readWavSamples, checkFfmpeg

  tts-engine.ts      -- TTSEngine class (lazy-loaded @animus-labs/tts-native PocketTTS, voice state caching)
  voice-manager.ts   -- VoiceManager (manifest CRUD, built-in discovery, custom uploads, WAV buffer loading)
  speech-service.ts  -- SpeechService facade + singleton (getSpeechService/initSpeechService)
  index.ts           -- barrel export

packages/tts-native/            -- napi-rs native addon wrapping pocket-tts Rust crate
  Cargo.toml                    -- Rust crate config (pocket-tts + napi-rs deps)
  build.rs                      -- napi-build setup
  src/lib.rs                    -- napi-rs bindings: PocketTTS + VoiceState classes
  package.json                  -- @animus-labs/tts-native npm package
  index.js                      -- Platform binary loader (auto-generated by napi-rs)
  index.d.ts                    -- TypeScript declarations
  tts-native.{platform}.node    -- Compiled native binary (build artifact, not committed)
```

### SpeechService Facade

The `SpeechService` is the public API. It wraps both engines and the voice manager behind a clean interface:

```typescript
class SpeechService {
  readonly stt: STTEngine;
  readonly tts: TTSEngine;
  readonly voices: VoiceManager;

  async initialize(): Promise<void>;
  isAvailable(): { stt: boolean; tts: boolean };
}

// Singleton access
function initSpeechService(): SpeechService;   // Called once at startup
function getSpeechService(): SpeechService;     // Used everywhere else
```

`initSpeechService()` is called in `packages/backend/src/index.ts` during server startup. Both engines independently lazy-load their models on first use -- `initSpeechService()` creates the service instance but does not load models. Models load on the first call to `stt.transcribe()` or `tts.synthesize()`.

### STTEngine

```typescript
class STTEngine {
  async transcribe(audio: Buffer | Float32Array, sampleRate?: number): Promise<string>;
  isAvailable(): boolean;  // Checks model files exist without loading
}
```

Lazy-loads the sherpa-onnx `OfflineRecognizer` on first `transcribe()` call. Uses dynamic `import('sherpa-onnx-node')` so the backend does not crash if the sherpa-onnx native module is not installed.

### TTSEngine

```typescript
class TTSEngine {
  async synthesize(text: string, options?: TTSSynthesisOptions): Promise<{
    samples: Float32Array;
    sampleRate: number;
    wavBuffer: Buffer;
  }>;
  isAvailable(): boolean;       // Checks safetensors model files exist without loading
  async ensureLoaded(): Promise<void>;  // Lazy-load the native PocketTTS model
  async setDefaultVoice(voiceId: string): Promise<void>;  // Cache reference audio
  dispose(): void;              // Release native resources
}

interface TTSSynthesisOptions {
  speed?: number;
  voiceId?: string;
}
```

Lazy-loads `@animus-labs/tts-native`'s `PocketTTS` model on first `synthesize()` call via dynamic `import('@animus-labs/tts-native')`. The native addon is compiled per-platform (requires Rust toolchain for first build). If the addon is not built, `ensureLoaded()` throws with a clear error message.

**Voice state caching**: The engine caches the most recently used `VoiceState` (the voice embedding computed from reference WAV audio). Creating a voice state involves running the WAV through the Mimi encoder and speaker projection matrix (~1s), so caching avoids repeating this on every synthesis call. The cache is invalidated when the voice ID changes.

### VoiceManager

Manages the voice manifest and WAV files:

```typescript
class VoiceManager {
  listVoices(): Voice[];
  getVoice(id: string): Voice | undefined;
  addCustomVoice(name: string, wavBuffer: Buffer): Promise<Voice>;
  removeCustomVoice(id: string): Promise<void>;
  discoverBuiltInVoices(): Voice[];
}

interface Voice {
  id: string;
  name: string;
  type: 'built-in' | 'custom';
  wavPath: string;         // Absolute path to reference WAV
  description?: string;
}
```

### Audio Utilities

```typescript
// Convert WebM/Opus to PCM (requires ffmpeg)
function webmToPcm(webmBuffer: Buffer): Promise<Float32Array>;

// Convert PCM samples to WAV buffer
function pcmToWav(samples: Float32Array, sampleRate: number): Buffer;

// Read WAV file and return samples
function readWavSamples(wavPath: string): Promise<{ samples: Float32Array; sampleRate: number }>;

// Check if ffmpeg is available
function checkFfmpeg(): Promise<boolean>;
```

- Saving to `data/voices/custom/`
- Updating the voice manifest
- Making the voice available for selection in the persona

### Voice in the Persona

The active voice is part of the persona configuration:

- `personality_settings.voice_id` -- ID of the selected voice (default: first built-in voice)
- `personality_settings.voice_speed` -- Speech speed multiplier (0.5-2.0, default: 1.0)

The TTSEngine reads the persona's voice settings when synthesizing. Changing the voice in Settings > Persona triggers `tts.setVoice()` to cache the new reference audio.

## Lazy Loading & Graceful Degradation

Both engines use dynamic imports and lazy initialization:

1. **No crash on missing models** -- If model files are not present, `isAvailable()` returns `false` and the service simply reports speech as unavailable. The rest of the backend runs normally.
2. **No crash on missing native modules** -- STT uses dynamic `import('sherpa-onnx-node')` and TTS uses dynamic `import('@animus-labs/tts-native')`. If either native addon is not built/installed, the dynamic import fails gracefully and that engine reports as unavailable.
3. **Auto-build for TTS addon** -- `npm run dev` runs `scripts/check-native.mjs` which checks for the `.node` binary and auto-builds it if Rust is installed. If Rust isn't available, it prints a warning and continues -- TTS is simply unavailable.
4. **Lazy model loading** -- Models are loaded into memory only on the first actual use (first `transcribe()` or `synthesize()` call), not at server startup. This keeps startup fast.
5. **Independent availability** -- STT and TTS load independently with different native addons. If only STT models are present, transcription works but synthesis does not (and vice versa).

```typescript
// Example: checking availability without loading
const speech = getSpeechService();
const { stt, tts } = speech.isAvailable();
// stt: true if sherpa-onnx model files exist on disk
// tts: true if safetensors model files exist on disk
// Neither engine is loaded yet -- just filesystem checks
```

## MCP Tools

Two MCP tools expose speech capabilities to the mind and sub-agents:

### `transcribe_audio`

- **Category**: safe
- **Permission**: always_allow
- **Input**: Audio file path or buffer
- **Output**: Transcribed text
- **Use case**: Mind or sub-agent needs to process audio content (e.g., a user uploaded an audio file via another channel)

### `generate_speech`

- **Category**: acts
- **Permission**: ask (produces audio output the user will hear)
- **Input**: Text to synthesize, optional voice_id and speed
- **Output**: WAV audio buffer or saved file path
- **Use case**: Mind decides to speak proactively, sub-agent needs to generate audio content

Both tools call through `getSpeechService()` and fail gracefully with a clear error message if the speech engine is not available.

## Consumer Integration

### Voice Channel Adapter

The voice channel adapter does **not** own STT or TTS instances. It calls the shared speech service:

```typescript
// In the voice channel adapter
const speech = getSpeechService();

// Transcribe incoming audio
const text = await speech.stt.transcribe(pcmAudio);

// Synthesize reply
const audio = await speech.tts.synthesize(replyText);
const wav = pcmToWav(audio.samples, audio.sampleRate);
```

See `docs/architecture/voice-channel.md` for the full voice channel architecture.

### Reflex System

The reflex system streams text chunks to the TTS for sentence-buffered synthesis:

```typescript
const speech = getSpeechService();

// Per-sentence synthesis during reflex streaming
const audio = await speech.tts.synthesize(sentence);
```

See `docs/architecture/reflex-system.md` for the dual-path voice architecture.

### Persona UI

Settings > Persona includes voice selection:
- Dropdown of available voices (built-in + custom)
- Upload button for custom voice WAV files
- Voice speed slider
- Preview/test button that synthesizes a sample sentence

## Error Handling

| Scenario | Behavior |
|----------|----------|
| sherpa-onnx not installed | STT engine reports unavailable. Backend runs normally. |
| @animus-labs/tts-native not built | TTS engine reports unavailable with message: "Native TTS addon not built. Run: npm run build -w @animus-labs/tts-native (requires Rust toolchain)". Backend runs normally. |
| Rust not installed | `scripts/check-native.mjs` prints warning on `npm run dev` and continues. TTS unavailable. |
| STT model files missing | `stt.isAvailable()` returns false. Transcription calls fail with clear error. |
| TTS model files missing | `tts.isAvailable()` returns false (checks for `tts_b6369a24.safetensors` + `tokenizer.model`). Synthesis calls fail with clear error. |
| ffmpeg not installed | `checkFfmpeg()` returns false. WebM-to-PCM conversion unavailable. Raw PCM input still works. |
| Invalid reference audio | `VoiceManager.addCustomVoice()` validates format and rejects with descriptive error. |
| Voice ID not found | `tts.synthesize()` falls back to default built-in voice with a warning log. |

## ONNX Pocket TTS Investigation

> **Status**: ONNX models produce garbled audio. Sticking with native Rust/Candle (safetensors).
> **Date**: Feb 2026
> **Tracked against**: `KevinAHM/pocket-tts-onnx` rev `4bd665d` (2026-02-06), initial ONNX release 2026-01-18

### What We Tested

We investigated whether ONNX-based Pocket TTS could replace our custom Rust/napi-rs binding (`@animus-labs/tts-native`). Three approaches were tested:

1. **sherpa-onnx-node 1.12.25** -- sherpa-onnx's native C++ runtime with KevinAHM's FP32 ONNX models, `numSteps=10`
2. **onnxruntime-node 1.21.0 direct** -- bypassing sherpa-onnx entirely, running the 5-model pipeline (text_conditioner → mimi_encoder → flow_lm_main → flow_lm_flow → mimi_decoder) directly via ONNX Runtime, ported 1:1 from KevinAHM's Python reference
3. **KevinAHM's original Python reference** (`pocket_tts_onnx.py`) -- the author's own inference code with onnxruntime 1.19.2

### Results

**All three approaches produced garbled, unintelligible audio.** This includes KevinAHM's own Python code running their own ONNX models with their own reference voice (`reference_sample.wav`). The garbling is consistent across:
- Different voice references (custom voices, KevinAHM's included reference)
- Different text inputs (short, long, multiple sentences)
- Different `numSteps` values (5 and 10)
- Different runtimes (sherpa-onnx C++, onnxruntime-node, Python onnxruntime)

### Root Cause

The issue is in the ONNX model export itself, not in any runtime or inference code. The `flow_lm_main.onnx` and `flow_lm_flow.onnx` models (the core flow-matching transformer) are exported at **opset 14**, while the decoder/encoder are opset 17-18. Something in the PyTorch-to-ONNX conversion for the flow-matching components loses numerical fidelity, producing corrupted latent representations that decode to garbled audio.

The original PyTorch safetensors weights (used by our Rust/Candle implementation) do not have this issue.

### Model Details (for future tracking)

| File | Size | Opset | Notes |
|------|------|-------|-------|
| `flow_lm_main.onnx` | 303 MB | 14 | Core transformer -- likely source of quality loss |
| `flow_lm_flow.onnx` | 39 MB | 14 | Flow network -- likely source of quality loss |
| `mimi_encoder.onnx` | 73 MB | 18 | Voice encoder |
| `mimi_decoder.onnx` | 42 MB | 17 | Audio decoder (stateful, 56 state tensors) |
| `text_conditioner.onnx` | 16 MB | 17 | Text embedding |

**Repo**: [huggingface.co/KevinAHM/pocket-tts-onnx](https://huggingface.co/KevinAHM/pocket-tts-onnx)
**Last checked revision**: `4bd665d8a6c8` (2026-02-06)
**Base model**: [kyutai/pocket-tts](https://huggingface.co/kyutai/pocket-tts)

### When to Re-evaluate

If KevinAHM updates the ONNX export (new commits after `4bd665d`), it's worth re-testing. An ONNX-based approach would be preferable since it eliminates the Rust toolchain build requirement and would allow using sherpa-onnx's streaming support. Check the HuggingFace repo for updates periodically.
